## What & why

Most of my side projects used to end up the same way: finished enough to matter, but buried in a private repo with most of the story missing.

Launch Start is my attempt to fix that structurally.

Each project gets a public sprint: a goal, a deadline, weekly write-ups, and a retrospective at the end. Instead of treating documentation as something I might remember to do later, Launch Start makes the trail part of the project from the beginning.

This site is the system that makes that sustainable.

Fittingly, it is also the first project on its own shelf.

The rule that shaped everything was simple: this has to survive a week where I only have 45 minutes on a Sunday.

So there is no CMS, no database, and no admin panel. Content lives as markdown in the repo, validated by schemas and rendered by templates. Writing a weekly log should be one scaffolded file, one honest update, and one push.

## Architecture

Launch Start is built around static content and boring reliability.

MDX content with zod-validated frontmatter goes in. Astro turns it into static HTML. Cloudflare serves it from the edge. The site rebuilds on every push, with a scheduled rebuild to keep sprint counters and time-based pages honest.

There are no runtime API calls, no deployed secrets, and no server that needs to stay alive for the site to work.

External facts, like GitHub and YouTube data, will be synced at build time as committed JSON snapshots. That means an API outage can make the data slightly stale, but it should never break a page.

The goal is not to make the fanciest possible system.

The goal is to make a system I will actually keep using.

## Outcome

Sprint 1 is the launch sprint.

The goal is to ship the full platform in 30 days: core pages, content schemas, project pages, weekly logs, CI checks, deployment, and a retrospective written after the work is done.

If it works, Launch Start becomes more than a portfolio. It becomes the place where my side projects get built, documented, and remembered.
